# Hangulpia 변환 엔진 가이드

한글(두벌식) → 일본어(히라가나) 변환 "모델"이 실제로 무엇이고, 어떻게 만들어지고,
어디서 어떻게 구동되는지 정리한 문서입니다. 딥러닝 모델이 아니라 **통계적 음소 정렬
테이블(PhoneticMap) + 손으로 정리한 문법 예외 규칙**으로 이루어져 있습니다.

- 코드 위치: `core/crates/db/src/phonetic_decoder.rs` (핵심 엔진)
- 관련 데이터: `core/crates/db/src/vocab.rs`, `vocab_extended.rs`, `vocab_large.rs`
- 실사용처: macOS 앱(`core/crates/macos-ime`), 홈페이지 데모(`homepage/index.html`)

## 1. 모델(PhoneticMap)을 생성하는 방법

### 1.1 이게 "학습된 모델"이 아닌 이유

이름은 통계적이지만 신경망 학습이 아닙니다. **결정론적으로 계산 가능한 역방향
테이블**을 만드는 것에 가깝습니다.

- 일본어 히라가나 → 한글 표기는 규칙만으로 결정됩니다 (`kana_hangul.rs`의
  `hiragana_to_hangul()` — つ→츠, ち→치 같은 고정 매핑).
- 그래서 (히라가나 읽기, 한글 표기) 쌍이 있으면, 그 반대 방향(한글→히라가나)의
  빈도 기반 확률 테이블을 **역산**할 수 있습니다. 이게 `PhoneticMap`입니다.

### 1.2 실제 빌드 코드

`core/crates/db/src/phonetic_decoder.rs`:

```rust
pub struct PhoneticMap {
    pub map: HashMap<String, Vec<PhoneticMapping>>, // hangul_token → 후보들
    pub total_pairs: u64,
}

impl PhoneticMap {
    pub fn build_from_pairs(&mut self, pairs: &[(String, String, u64)]) {
        // pairs = (hiragana_reading, hangul, frequency)
        // align_hiragana_hangul()로 문자 단위 정렬 후 빈도 카운트 → 확률로 정규화
    }
}
```

`align_hiragana_hangul()`은 ん(받침 처리)/っ(촉음)/요음(2글자 히라가나 ↔ 1글자
한글) 같은 비1:1 매칭까지 처리합니다.

### 1.3 실제로 모델을 만드는 명령

```bash
cd core

# 방법 A — 앱이 매번 부팅 시 하는 것과 동일 (in-memory, 파일 없음)
#   crates/macos-ime/src/ffi.rs::hj_engine_init() 참고

# 방법 B — CLI로 SQLite에 영구 저장 (연구/실험용, 실제 앱은 이 파일을 안 읽음)
cargo run -p ime-cli --bin hj-ime -- db-build-large my-model.db --count 100000
```

`db-build-large`는 `generator.rs`의 템플릿 기반으로 **합성 문장**을 생성해 학습
쌍을 만듭니다 (실제 위키피디아 원문이 아님 — 1.4절 참고).

### 1.4 어휘 소스 3종 (2026-07-23 기준, v0.1.3)

| 소스 | 함수 | 항목 수 | 성격 |
|---|---|---|---|
| `vocab.rs` | `build_vocab()` | 731 | 손으로 큐레이션한 인사말/가족/기초 단어 |
| `vocab_extended.rs` | `build_extended_vocab()` | 711 | 확장 어휘 |
| `vocab_large.rs` | `build_vocab_large()` | 22,741 | `gen_vocab_large.py`로 자동 생성 (스크립트 자체는 저장소에 없음) |
| **합계** | `build_vocab()+build_extended_vocab()+build_vocab_large()` | **24,914** | v0.1.3부터 실제 앱이 이걸 사용 |

**⚠️ 중요한 함정 (직접 검증됨)**: 저장소에 있던 `data/hj-ime-large.db`(2.8MB,
CHANGELOG에 "11만 문장 위키피디아 코퍼스"로 기록됨)는 **실제 앱 초기화 경로에
연결된 적이 없습니다.** `hj_engine_init()`은 항상 위 표의 어휘를 즉석에서
새로 빌드합니다. 그 DB 파일은 연구용 산출물로만 존재했고, 지금은 로컬에서
빈 파일(0바이트)로 확인됨 — git에도 추적된 적 없음(`*.db`는 `.gitignore`
대상). 진짜 11만 문장 코퍼스 원본(191MB)도 저장소에 없습니다.

### 1.5 예외 규칙 (통계로 안 잡히는 것들)

`PhoneticMap`은 문자 정렬 통계라서 **문법**을 모릅니다. 조사 は(발음 wa)나
です/ます 활용형처럼 "문맥에 따라 다르게 써야 하는" 패턴은 학습으로 안 풀려서,
`phonetic_decoder.rs`에 하드코딩된 예외 테이블로 별도 처리합니다:

```rust
const KNOWN_WORDS: &[(&str, &str)] = &[
    ("사쿠라", "さくら"), ("아리가토", "ありがとう"), ("곤니찌와", "こんにちわ"),
    /* ... */
];

const KNOWN_SUFFIXES: &[(&str, &str)] = &[
    ("데스카", "ですか"), ("데스", "です"), ("마스", "ます"),
    ("와", "は"), // 조사, 반드시 맨 뒤(최단 매치)에 위치
];
```

새 예외를 추가하고 싶으면 이 두 배열에 추가 + `homepage/index.html`의 동일한
이름의 JS 객체(`KNOWN_WORDS`/`KNOWN_SUFFIXES`)에도 **반드시 동기화**해야 합니다
(3절 참고).

## 2. 모델을 이용하는 방법 (Rust API)

### 2.1 최소 예시

```rust
use ime_db::phonetic_decoder::{PhoneticMap, BeamDecoder};
use ime_db::kana_hangul::hiragana_to_hangul;
use ime_db::vocab::build_vocab;
use ime_db::vocab_extended::build_extended_vocab;
use ime_db::vocab_large::build_vocab_large;

let mut vocab = build_vocab();
vocab.extend(build_extended_vocab());
vocab.extend(build_vocab_large());

let pairs: Vec<(String, String, u64)> = vocab.iter()
    .map(|v| (v.reading.to_string(), hiragana_to_hangul(v.reading), 100u64))
    .collect();

let mut map = PhoneticMap::new();
map.build_from_pairs(&pairs);

let decoder = BeamDecoder::new(&map, /* beam_width */ 6, /* max_candidates */ 5);

// 추천 API — 문장 단위, 예외 규칙 자동 적용, 최선 후보 1개만 반환
let result: String = decoder.decode_sentence("나마에와 난데스카");
// => "なまえは なんですか"

// 원시 API — 후보 여러 개 + 신뢰도 (단어 하나 단위, 예외 규칙 미적용)
let candidates: Vec<(String, f64)> = decoder.decode("사쿠라");
// => [("さくら", 0.97..), ...]
```

`decode_sentence()`가 실제 앱과 동일한 결과를 주는 진입점입니다.
`decode()`를 직접 쓸 거면 조사/활용형 예외가 안 걸린다는 점을 기억하세요.

### 2.2 CLI로 써보기

```bash
cd core
cargo run -p ime-cli --bin hj-ime -- "사쿠라"
cargo run -p ime-cli --bin hj-ime -- --interactive
```

⚠️ 이 기본 CLI 경로(`crates/cli/src/main.rs`)는 **아직 위 `BeamDecoder`가 아니라
더 오래된 규칙 기반 파이프라인**(`ime_hangul::phoneme` + `ime_japanese::romaji`)을

example을 짜서 돌리거나, `--db` 옵션으로 미리 만든 DB를 지정해야 합니다.

### 2.3 테스트

```bash

```

`곤니찌와`, `와`(조사), `데스카` 등 알려진 함정 케이스가 회귀 테스트로 고정되어
있습니다 (`phonetic_decoder.rs` 하단 `mod tests`).

## 3. 웹페이지에서 모델을 이용하는 방법 (구동·연동 메커니즘)

### 3.1 요약

홈페이지(`homepage/index.html`)의 "Try it yourself" / "sentence mode" 데모는
**진짜 24,914개 PhoneticMap을 쓰지 않습니다.** 그 데이터를 브라우저로 보내는 건
비현실적이라, `phonetic_decoder.rs`의 **규칙 기반 fallback 경로**
(`hangul_char_to_hiragana_fallback` + `romaji_to_hiragana_simple`)와 **예외
테이블**(`KNOWN_WORDS`/`KNOWN_SUFFIXES`)을 그대로 JavaScript로 손으로 포팅해서
씁니다.

```
core/crates/db/src/phonetic_decoder.rs   (Rust, 정답 소스)
        │  사람이 손으로 동일 로직을 JS로 옮김 (자동 변환 아님)
        ▼
homepage/index.html <script> 안의 IIFE   (JS, 브라우저에서 실행)
```

### 3.2 구동 메커니즘 (런타임)

1. 정적 HTML/CSS/JS 한 파일. 빌드 스텝도, 서버도, API 호출도 없습니다.
2. 페이지 로드 시 즉시 실행되는 `(function () { ... })()` 블록 하나가:
   - 자모 분해 테이블(`CHOSEONG_MAP`, `JUNGSEONG_MAP`), 로마자→히라가나 표
     (`HIRA_TABLE`), 예외 사전(`KNOWN_WORDS`, `KNOWN_SUFFIXES`)을 정의
   - `wire(inputId, outputId)` 함수로 두 개의 입력창을 연결:
     - `#tryInput` → `#mockOutput` (짧은 단어 데모)
     - `#playgroundInput` → `#playgroundOutput` (긴 문장 연습)
   - 각 입력창의 `input` 이벤트마다 `hangulToHiragana(value)`를 호출해 실시간 렌더링
3. `hangulToHiragana()`는 공백 기준으로 단어를 쪼갠 뒤 각 단어를
   `convertWord()`에 넘기고, `convertWord()`는:
   1. `KNOWN_WORDS` 정확히 일치 → 그 값 반환
   2. `KNOWN_SUFFIXES` 중 가장 긴 것부터 접미사 매치 → 어간은 재귀 처리 + 어미는 고정값
   3. 둘 다 아니면 글자 하나씩 `hangulCharToHiraganaFallback()`으로 분해 (자모 → 로마자 후보들 → 확률 최고값 픽)

### 3.3 실제 엔진과의 차이 (정직하게)

| | Rust 실제 엔진 | 홈페이지 JS 데모 |
|---|---|---|
| 어휘 규모 | 24,914개 학습 | 10개 하드코딩 |
| 통계적 확률 결합 | O (BeamDecoder 빔서치) | X (단순 최고 확률 1개만) |
| 조사/활용형 예외 | O | O (동일 목록 수동 동기화) |
| 실행 위치 | 사용자 기기의 네이티브 프로세스 | 사용자 브라우저 (JS) |

`KNOWN_WORDS`/`KNOWN_SUFFIXES`를 한쪽만 고치면 데모와 실제 앱 결과가 어긋나므로
**항상 양쪽 다 수정**하세요:
- `core/crates/db/src/phonetic_decoder.rs` (Rust `const` 배열)
- `homepage/index.html` (JS 객체/배열, 같은 이름)

## 4. 다운로드해서 이용하는 방법 및 구동 메커니즘 (macOS 앱)

### 4.1 사용자 입장에서

```bash
# hangulpia.com에서 HangulJapaneseIME-latest.zip 다운로드 후
unzip HangulJapaneseIME-latest.zip
cd HangulJapaneseIME-v0.1.3-arm64
bash install.sh
# 로그아웃 → 재로그인
# 시스템 설정 → 키보드 → 입력 소스 편집 → + → 일본어 카테고리 → 한글일본어입력기
```

- 요구사항: macOS 13+ (Ventura), Apple Silicon(arm64) 전용 — Intel Mac/Windows 미지원
  (Windows 소스는 `core/crates/windows-ime`에 있지만 빌드된 배포판 없음)
- 제거: `bash uninstall.sh`

### 4.2 아키텍처

```
┌─────────────────────────────────────────┐
│  Swift (InputMethodKit 앱)                │
│  HJInputController.swift  ← 키 입력 수신   │
│  HJEngine.swift           ← FFI 래퍼      │
└───────────────┬───────────────────────────┘
                │ C FFI (extern "C")
                ▼
┌─────────────────────────────────────────┐
│  Rust (libhj_engine.a, 정적 라이브러리)    │
│  crates/macos-ime/src/ffi.rs              │
│    hj_engine_init()                       │
│    hj_hangul_to_hiragana()  ← 실시간 조합용│
│    hj_convert()             ← 후보창용     │
└─────────────────────────────────────────┘
```

### 4.3 구동 메커니즘 (런타임)

1. **앱 시작 시** (`HJEngine.shared.initialize()` → `hj_engine_init()`):
   - 24,914개 전체 어휘(1.4절)로 `PhoneticMap`을 **매번 새로 빌드** (파일 저장/로드 없음, 순수 in-memory)
   - 칸지 사전도 같은 어휘로 in-memory SQLite(`:memory:`)에 적재
2. **타이핑 중** (실시간 조합 텍스트):
   - `HJInputController`가 두벌식 자모를 `hangulBuffer`에 누적
   - 매 키 입력마다 `HJEngine.shared.toHiragana()` → `hj_hangul_to_hiragana()`
     → `BeamDecoder::decode_sentence()` (v0.1.3부터 조사/활용형 예외 규칙 포함)
   - 결과를 조합 중 텍스트로 화면에 표시
3. **후보 선택** (숫자 1-9로 대체 후보 고를 때):
   - `HJEngine.shared.convert()` → `hj_convert()` → `BeamDecoder::decode()`로
     후보 여러 개 생성 → 각 후보로 칸지 사전 조회 → 점수순 정렬해 후보창에 표시
4. **확정** (스페이스/엔터): 조합 중이던 히라가나(또는 선택한 칸지 후보)가 커밋됨

### 4.4 빌드 메커니즘 (개발자용)

```bash
cd core/crates/macos-ime
./build.sh build      # 빌드만
./build.sh install    # 빌드 + ~/Library/Input Methods/ 에 설치 + 코드사인
./dist.sh              # 빌드 + 배포용 zip 생성 (dist/HangulJapaneseIME-vX.Y.Z-arm64.zip)
```

`build.sh` 내부:
1. `cargo build --release -p ime-macos --target aarch64-apple-darwin`
   → `target/aarch64-apple-darwin/release/libhj_engine.a`
2. `swiftc`로 Swift 소스들을 컴파일하면서 위 정적 라이브러리를 링크
   (`-framework Cocoa -framework Carbon -framework InputMethodKit -framework Security`)
3. `.app` 번들 조립 (`Info.plist`, 메뉴바 아이콘 생성, 실행 파일)

`dist.sh`는 `build.sh build`를 호출한 뒤 `.app` + `install.sh`/`uninstall.sh` +
README를 묶어 zip으로 패키징합니다. **버전을 올릴 땐 아래 세 곳을 다 맞춰야
합니다** (안 그러면 zip 파일명과 앱 내부 버전이 어긋남):

- `core/Cargo.toml` → `[workspace.package] version`
- `core/crates/macos-ime/dist.sh` → `VERSION=` 및 안내 문구 속 버전 문자열들
- `core/crates/macos-ime/HangulJapaneseIME/Resources/Info.plist` → `CFBundleShortVersionString`/`CFBundleVersion`

### 4.5 알려진 한계

- 탁음 모호성(が/か, ば/ぱ 등)은 통계 확률이 완벽히 못 잡음 — 한글 초성 하나가
  일본어 청음/탁음 둘 다에 대응되는 경우가 많아서 구조적으로 남는 문제
- 장음(長音) 누락 — 한글 표기 자체에 장음을 나타낼 방법이 마땅치 않아 「ありがとう」의
  「う」 같은 게 학습 데이터에 없으면 빠질 수 있음
- `KNOWN_WORDS`/`KNOWN_SUFFIXES`는 손으로 짠 유한 목록이라 다루지 않는 문법
  패턴은 여전히 많음 (조사 へ/を 등은 아직 미대응 — 표준 대응 후보 대비 빗나갈
  때가 더 많아서 일부러 안 넣음)
- Windows 빌드 없음, Intel Mac 미지원

#### 1. 서론

일본어 텍스트 입력 시스템은 광범위한 연구와 상업적 개발의 대상이 되어왔다.
지배적인 방식인 로마지 입력은 사용자가 일본어 발음의 로마자 표현을
입력하면(예: "arigatou" → ありがとう) 히라가나로 변환한 후 사전 검색과 언어
모델링을 통해 한자로 변환하는 방식이다. 본 논문은 대안적 접근법으로 한글
키보드 배치를 일본어 음소 입력 매체로 활용하는 방법을 탐구한다.

한글은 일본어 음소 입력에 독특한 이점을 제공한다. 한글은 초성-중성-종성
음소를 하나의 콤팩트한 음절 블록으로 인코딩하는 자질 문자(featural
alphabet)이다. 하나의 한글 음절 블록은 일본어 모라(mora) 하나를 1~2번의 키
입력으로 표현할 수 있어, 로마지의 2~4번과 비교하면 효율적이다. 예를 들어,
일본어 か(ka)는 로마지로 k+a 2키가 필요하지만, 한글 두벌식 배치에서는
ᄏ+ᅡ로 동일하게 2키이면서도 더 복잡한 음절에서는 큰 차이를 보인다.

**1.1 문제 정의**

한글→일본어 변환 파이프라인은 표준 일본어 IME와 구별되는 여러 기술적 과제에
직면한다. 첫째, 한국어와 일본어 간의 음소 매핑이 본질적으로 모호하다: 여러
히라가나 시퀀스가 동일한 한글 표현을 생성할 수 있다(예: つ(tsu)와
ちゅ(chyu)가 유사한 한글로 매핑). 둘째, 일본어 장음(예: こう kou vs. こお
koo)이 한글 영역에서 추가적인 모호성을 생성한다. 셋째, 히라가나 후보 자체가
확률적이므로 기존 IME보다 훨씬 더 큰 후보 공간에서 한자 판별을 수행해야 한다.

**1.2 기여**

본 논문의 기여는 다음과 같다: (1) 한글 입력에서 히라가나 Top-1 정확도
90.5%를 달성한 음소 임베딩 모델(규칙 기반 대비 +30.9%p 개선); (2) 한자 판별
정확도를 +25.7%p 향상시킨 인접문장 공출현 임베딩 방법; (3) 음소 신뢰도, 문맥
유사도, 빈도, N-gram 점수를 결합하여 62.1% Top-1 표기 정확도를 달성한 4요소
랭킹 공식; (4) 학습량 스케일링, 어휘 확장, 윈도우 크기, 방향성 임베딩, 장음
분석 등 14개 차원에 걸친 포괄적 실험 평가; (5) KSPC 0.85의 macOS 및 Windows
네이티브 IME 구현.

#### 2. 시스템 구조

본 IME 시스템은 Rust 워크스페이스로 구현되어 있으며, core, hangul, japanese,
db, macos-ime, windows-ime, android-bridge, cli 등 8개의 크레이트로 구성된다.
변환 파이프라인은 입력되는 각 단어에 대해 4단계로 순차 실행된다.

**2.1 1단계: QWERTY → 한글 자모 조합**

물리적 키보드 이벤트(QWERTY 키코드)를 먼저 표준 두벌식 배치를 사용하여
한글 자모(자음과 모음)로 매핑한다. 자모는 유니코드 조합 공식(codepoint =
0xAC00 + cho×21×28 + jung×28 + jong)을 사용하여 한글 음절 블록으로
조합된다. 복합 모음(ㅗ+ㅡ→ㅘ)과 복합 자음(ㄱ+ㅅ→ㄳ)을 상태 머신으로 처리한다.

**2.2 2단계: 한글 → 히라가나 디코딩**

조합된 한글 문자열은 학습된 PhoneticMap과 BeamDecoder를 사용하여 히라가나
후보로 변환된다. PhoneticMap은 500K 학습 문장 쌍에서 학습된 통계 모델로,
각 한글 토큰을 확률 값이 부여된 하나 이상의 히라가나 시퀀스로 매핑한다.
BeamDecoder는 빔 폭 8, 최대 후보 20개로 빔 검색을 수행한다.

**2.3 3단계: 한자 후보 검색**

각 히라가나 후보는 한자 사전(KanjiDict)에서 가능한 모든 표기형을 검색한다.
예를 들어 히라가나 こうえん은 公園(공원), 講演(강연), 公演(공연), 後援(후원)
등의 후보로 매핑된다. 각 후보에는 학습 코퍼스에서의 빈도 정보가 포함된다.

**2.4 4단계: 문맥 인식 랭킹**

후보들은 4요소 가중 공식으로 점수화된다: `score = α·phoneme + β·context +
γ·frequency + δ·ngram` (α=0.1, β=0.35, γ=0.25, δ=0.3). 문맥 점수는 후보의
64차원 단어 임베딩과 이전에 확정된 단어들의 가중 평균 임베딩 간의 코사인
유사도로 계산된다. N-gram 점수는 바이그램/트라이그램 조건부 확률로 순차적
패턴을 포착한다.

#### 3. 실험 방법론

**3.1 코퍼스 생성**

모든 실험은 표기형, 읽기(히라가나), 한글 음역이 포함된 문장을 생성하는
합성 일본어 코퍼스 생성기를 사용한다. 생성기는 기본 어휘 1,442개 또는 확장
어휘 23,472개에서 2~5개 단어로 문법적으로 타당한 문장을 구성한다. 학습/테스트
세트는 인덱스 오프셋으로 분리하여 데이터 누출을 방지한다.

**3.2 평가 지표**

모든 실험에서 다음 지표를 사용한다: Top-1 표기 정확도(최상위 후보가 원본
표기형과 일치), 히라가나 Top-1(정답 읽기가 최상위 음소 디코드), 히라가나
Any-Hit(정답이 후보 목록 어디에든 포함), Top-K Hit Rate(K=3,5,10), 문서 전체
일치(문장 내 모든 단어 정확 변환), KSPC(물리적 키 입력 수 / 출력 문자 수).

**3.3 학습 구성**

별도 명시가 없는 한, 모든 실험은 다음 기본 구성을 사용한다: 학습 문장
500,000개, 64차원 단어 임베딩(SGD 50회 반복), 공출현 윈도우 크기 5, 테스트
문서 1,000건(계산 집약적 실험은 300건). 모든 벤치마크는 디스크 I/O 변동을
제거하기 위해 인메모리 SQLite 데이터베이스에서 실행한다.

#### 4. 실험 결과

**4.1 음소 디코딩: 규칙 기반 vs. 임베딩**

첫 번째 실험은 한글을 히라가나로 변환하는 두 가지 접근법을 비교한다: (A)
한글→로마지→히라가나 결정론적 규칙 파이프라인, (B) PhoneticMap + BeamDecoder
통계적 매핑. 양쪽 모두 동일한 문맥 랭킹을 사용한다.

| 지표 | 규칙 기반 (A) | 음소 임베딩 (B) | Δ (B−A) |
|---|---|---|---|
| 히라가나 Top-1 | 59.6% | 90.5% | +30.9%p |
| 히라가나 Any-Hit | 89.2% | 95.0% | +5.8%p |
| Top-1 표기 | 17.8% | 27.1% | +9.3%p |
| 읽기 일치 | 63.6% | 90.5% | +26.8%p |
| Top-10 Hit | 87.7% | 94.6% | +6.8%p |
| 문서 전체 일치 | 2.3% | 5.2% | +2.9%p |

음소 임베딩 접근법은 히라가나 Top-1 정확도에서 +30.9%p의 극적인 개선을
달성하였다(59.6% → 90.5%). PhoneticMap은 224개의 고유 한글 토큰과 266개의
매핑을 학습하였으며(토큰당 평균 모호성 1.2개), 500K에서 10M 학습량까지
변화가 없어 음소 매핑 공간이 조기에 포화됨을 보여준다.

**4.2 마커 및 멀티토큰 음소 디코딩**

기본 음소 디코더에 대한 3가지 개선을 테스트하였다: (1) 모호한 음소 경계에
특수 기호를 삽입하는 마커 방식, (2) BeamDecoder가 단일 문자와 함께 2~3문자
한글 토큰을 고려하는 멀티토큰 방식, (3) 둘을 결합한 하이브리드. 멀티토큰
방식은 Top-10 상한을 94.5%에서 97.6%로 높였다(+3.1%p).

**4.3 인접문장 공출현 임베딩**

표준 단어 임베딩은 문장 내 공출현 패턴만 포착한다. 인접문장 공출현은
인접한 문장의 단어도 공출현 행렬에 기여하게 하여(cross_weight 가중치 적용)
더 넓은 주제적 일관성을 포착한다.

인접문장 문맥은 단일 최대 개선 요소로, Top-1 표기 정확도에 +25.7%p를

**4.5 N-gram 언어 모델 통합**

랭킹 공식에 바이그램/트라이그램 조건부 확률을 추가하였다. N-gram 모델은
순차적 패턴을 포착한다(예: P(天気|今日,の)가 높아 "오늘의" 뒤에 날씨 관련
완성을 편향).

| 구성 | α | β | γ | δ | Top-1 표기 | Δ |
|---|---|---|---|---|---|---|
| 기준 (N-gram 없음) | 0.1 | 0.5 | 0.4 | 0.0 | 49.4% | — |
| δ=0.1 | 0.1 | 0.5 | 0.4 | 0.1 | 52.8% | +3.4%p |
| δ=0.2 (−β) | 0.1 | 0.3 | 0.4 | 0.2 | 55.5% | +6.1%p |
| δ=0.3 (−βγ) 최적 | 0.1 | 0.35 | 0.25 | 0.3 | 56.1% | +6.7%p |
| δ=0.4 (−βγ) | 0.1 | 0.25 | 0.25 | 0.4 | 55.1% | +5.7%p |

*표 4. N-gram 언어 모델 가중치 스위프 (300건 테스트)*

N-gram 모델은 순 +6.7%p의 개선을 제공한다. 최적 가중치 분배(α=0.1, β=0.35,
γ=0.25, δ=0.3)는 문맥 임베딩(β)과 빈도(γ) 모두에서 균등하게 가져왔으며,
이는 N-gram이 두 신호를 부분적으로 포섭함을 반영한다.

| 2M | 43.1% | 62.7% | +19.6%p |

*표 5. 학습량 스케일링: 기존 vs. 최적화 파이프라인*

핵심 발견은 최적화 구성에서 500K 학습(61.7%)이 기존 방법 2M 학습(43.1%)을
18.6%p 초과한다는 점이다. 500K→2M에서 불과 +1.0%p 개선으로, 500K가 비용
대비 효과의 최적점임을 확인하였다.

| 기본 어휘 + 최적 구성 | 1,442 | 62.0% | 96.3% |
| 확장 어휘 + 기본 가중치 | 23,472 | 28.2% | — |
| 확장 어휘 + 최적 구성 | 23,472 | 39.8% | 81.7% |

*표 6. 어휘 확장 A/B 테스트 (500K 학습, 1000건 테스트)*

어휘 확장은 Top-1 정확도를 62.0%에서 39.8%로 급격히 떨어뜨렸다(−22.2%p).
근본 원인은 동음이의어 폭발: 학습 데이터를 비례적으로 확장하지 않고
어휘만 확장하면 히라가나 읽기당 한자 후보가 폭발적으로 증가한다. 히라가나
디코딩 자체는 개선되지만(92.6% → 94.4%), 하류 한자 선택이 부족한 공출현
데이터 밀도로 인해 실패한다.

> **연결 노트**: 이 4.7절 실험은 흥미로운 대조를 이룹니다 — 이 논문에서는
> "어휘를 23,472개로 늘렸더니 정확도가 떨어졌다"(한자 판별 단계에서),
> 반대로 실제 배포 앱(v0.1.3)에서는 "어휘를 731→24,914개로 늘려서
> 조사/활용형 커버리지를 넓혔다"(1.4절). 모순처럼 보이지만 실은 **다른
> 문제를 다룬 것**입니다 — 이 논문의 실험은 *한자 판별* 단계의 동음이의어
> 문제이고, 실제 앱의 PhoneticMap 확장은 *한글→히라가나 정렬 학습 데이터*
> 확보 목적입니다. 실제 앱은 한자 판별 자체를 라이브 타이핑 경로에서
> 아예 안 씁니다 (6절 참고).

**4.9 방향성 vs. 무방향 임베딩**

대칭(무방향) 공출현 임베딩과 전방향/후방향 별도 벡터를 유지하는 방향성
임베딩(2배 저장 공간 필요)을 비교하였다. 결과는 두 접근법 간 무시할 수 있는
차이를 보여, 거리 가중치(1/d) 공출현이 명시적 방향성 없이도 충분한 위치
정보를 포착함을 확인하였다.

**4.10 장음 마커 분석**

일본어 장음(예: こう /kou/, せい /sei/)은 한글 표현에서 상당한 모호성을
생성한다. 두 번째 모음(う 또는 い)이 별도 한글 문자로 매핑되기 때문이다.
한글 입력 스트림에 틸드(~) 마커를 삽입하여 장음을 명시적으로 표시하는
방법을 테스트하였다.

| 지표 | 기존 한글 | ~ 마커 적용 | 변화 |
|---|---|---|---|
| 히라가나 Hit | 96.3% | 74.0% | −22.3%p |
| Top-1 표기 | 61.6% | 48.7% | −12.8%p |
| Top-3 표기 | 85.2% | 66.2% | −19.1%p |
| 장음 단어 히라가나 Hit | 97.8% | 51.7% | −46.1%p |

*표 8. 장음 마커(~) 영향 분석 (300건 테스트, 최적 구성)*

장음 마커는 모든 지표에서 정확도를 급격히 저하시켰다. 히라가나 적중률이
22.3%p 하락했으며, 장음 포함 단어에서는 46.1%p의 치명적 하락을 보였다.
근본 원인은: (1) ~ 토큰이 PhoneticMap 학습 코퍼스에 부재, (2) 가타카나
장음 부호(ー) 변환 실패, (3) 추가 입력 부담이 이론적 모호성 해소 이익을
초과. 명시적 장음 마커 사용을 비추천하며, 문맥 기반 후처리를 권장한다.

> **연결 노트**: 실제 배포 앱에서도 장음 누락은 여전히 미해결 상태입니다
> (4.5절 "알려진 한계" 참고 — 아리가토 → ありがとう의 う가 종종 빠짐).
> 이 논문의 결론("문맥 기반 후처리를 권장")대로, 우리는 최종적으로
> `KNOWN_WORDS` 사전에 "아리가토"→"ありがとう" 같은 정답을 직접 박아넣는
> 방식으로 우회했습니다 — 이것도 넓게 보면 이 논문이 권장한 "후처리"의
> 한 형태입니다.

#### 6. 플랫폼 구현

**6.1 macOS (InputMethodKit)**

macOS 구현은 Apple의 InputMethodKit 프레임워크를 사용하며, Rust 코어
라이브러리를 정적 라이브러리(libhj_engine.a)로 컴파일하고 C FFI 레이어를
통해 Swift로 브릿징한다. TIS(Text Input Source) API로 시스템 입력 소스로
등록하고, IMKTextInput 프로토콜로 인라인 구성을 렌더링한다. 주요 구현
과제로는 Info.plist 구성, 설치 후 로그아웃/로그인 필요성, 메뉴 바 배지
아이콘(韓あ) 생성 등이 있었다.

**6.2 Windows (키보드 훅)**

Windows 구현은 WH_KEYBOARD_LL 저수준 키보드 훅으로 키 입력을 인터셉트하고,
KEYEVENTF_UNICODE 플래그의 SendInput으로 변환된 문자를 주입한다.
Ctrl+Space로 토글하며, 플랫폼 의존성 없이 순수 Rust로 한글 자모 조합을
구현하였다. 복잡한 TSF(Text Services Framework) 대신 키보드 훅 방식을
선택하여 구현 단순성과 Windows 10/11 호환성을 확보하였다.

#### 7. 논의

**7.1 주요 발견**

세 가지 발견이 돋보인다. 첫째, 음소 임베딩 단계가 빠르게 포화하여(224개
토큰, 500K~10M까지 불변), 한글-일본어 음소 매핑 공간이 본질적으로
제한적이며 병목이 전적으로 하류 한자 판별에 있음을 시사한다. 둘째,
인접문장 공출현 임베딩이 단일 최대 정확도 개선(+25.7%p)을 제공하여, 문장
경계를 넘는 주제적 일관성이 일본어 단어 예측에 강력한 신호임을 보여준다.
셋째, 비례적 학습 데이터 확장 없는 어휘 확장이 역효과적이라는 발견은
시스템 어휘 관리 전략에 직접적인 시사점을 준다.

**7.2 한계**

상용 IME와의 32.9%p 정확도 격차는 여전히 상당하다. 주요 한계는 확률적
히라가나 디코딩 단계로, 상용 IME는 결정론적 히라가나 입력을 받아 오류
원천 자체를 제거한다. 또한 합성 학습 코퍼스가 실제 일본어 텍스트의
복잡성을 완전히 포착하지 못할 수 있다. 명사 판별 병목(48.4%)은 범용
임베딩을 넘어서는 도메인 특화 솔루션이 필요하다.

**7.3 향후 연구**

상용 IME 수준으로 정확도를 향상시킬 수 있는 방향: (1) 실제 일본어 텍스트
코퍼스(예: 위키피디아, 뉴스 기사) 통합; (2) 명사 판별을 위한 의미 카테고리
제약 구현; (3) 문맥 랭킹을 위한 트랜스포머 기반 언어 모델 탐색; (4) 다음
단어 예측을 0.6%에서 사용 가능 수준으로 개선; (5) 확정된 선택으로부터
학습하는 사용자 적응.

#### 8. 결론

본 논문은 Top-1 표기 정확도 62.1%, KSPC 0.85를 달성한 완전한
한글→일본어 입력기 엔진을 제시하였다. 14개의 체계적 실험을 통해 크로스
링귀스틱 IME 정확도의 핵심 요소를 식별하였다: 음소 임베딩 품질, 문장 간
의미 문맥, N-gram 언어 모델링, 적절한 어휘-학습데이터 비율. 인접문장
공출현 임베딩이 단독으로 +25.7%p를 기여하여 가장 영향력 있는 아키텍처
혁신이었다. 본 시스템은 한글 음소 인코딩이 기존 로마지보다 근본적으로 더
효율적인 일본어 입력 매체임을 입증하였으며, 문자당 키 입력을 57.5%
절감하면서 사용 가능한 변환 정확도를 유지한다. macOS 및 Windows 네이티브
구현으로 실용적 배포 가능성을 확인하였다.

## 6. + 알파 — 이 논문과 실제 배포 앱의 관계

논문 전문을 그대로 옮겨두긴 했지만, **1~4절에서 설명한 실제 배포 중인
macOS 앱(v0.1.3)은 이 논문이 설명하는 시스템과 완전히 같지 않습니다.**
헷갈리지 않도록 정확한 관계를 정리합니다.

### 6.1 무엇이 겹치고 무엇이 안 겹치는가

| 논문이 설명하는 구성 요소 | 코드 위치 | 실제 앱 라이브 타이핑 경로(`hj_hangul_to_hiragana`)에 연결됨? |
|---|---|---|
| PhoneticMap + BeamDecoder (한글→히라가나) | `phonetic_decoder.rs` | ✅ 예 — 다만 4장 실험은 **합성 코퍼스 500K 문장** 기준, 실제 앱은 **어휘 24,914개**(3.3절 규모와 다름) |
| 음소 임베딩(64차원 단어 벡터) | `embedding.rs` | ❌ 아니오 — 조사/활용형 예외 규칙(1.5절)으로 대체됨 |
| 인접문장 공출현 임베딩 | `sentence.rs` (`SentenceBuffer`) | ❌ 아니오 — 라이브 타이핑은 단어 단위 처리, 문장 간 문맥 참조 없음 |
| N-gram 언어 모델 | `ngram.rs` | ❌ 아니오 |
| 한자 판별(KanjiDict + ContextRanker) | `dictionary.rs` | ⚠️ 부분적 — `hj_convert()`(후보창용)만 사용, 기본 조합 텍스트 경로(`hj_hangul_to_hiragana`)는 히라가나까지만 감 |
| 자동 완성 | `autocomplete.rs` | ❌ 미배선 (CLI 벤치마크에만 존재) |

즉, **논문은 "만약 한자 판별까지 하는 풀 파이프라인을 붙이면 어떻게 될까"를
실험한 연구 기록**이고, 실제 배포된 v0.1.3은 그중 **히라가나 변환 단계만
떼어내어 제품화**한 것입니다. 한자 후보/문맥 랭킹/N-gram은 코드로는
존재하지만 사용자가 타이핑할 때 실제로 호출되는 경로에는 없습니다.

### 6.2 정확도 수치를 그대로 믿으면 안 되는 이유

- 논문의 62.1%/90.5% 등은 **합성 생성 코퍼스**(실제 사람 문장이 아니라
  `generator.rs` 템플릿으로 만든 문장) 기준입니다. 4.7절 스스로도 밝히듯
  이런 수치는 어휘·학습량 비율에 민감합니다.
- 반면 이 문서 4.5절 "알려진 한계"와 세션 중 실제로 검증한 사례(`아나타노
  나마에와 난데스카`, `곤니찌와` 등, `phonetic_decoder.rs`의
  `test_wa_particle_is_ha_standalone_and_attached` 등 회귀 테스트로 고정됨)는
  **실제 프로덕션 코드 경로(24,914 어휘 + BeamDecoder)를 사람이 손으로 만든
  실제 문장에 직접 돌려서 발견한 결과**입니다. 두 수치 체계는 같은 잣대가
  아니므로 직접 비교하지 마세요.

### 6.3 이 논문에서 아직 살리지 않은 아이디어 (향후 개선 후보)

- **인접문장 공출현 임베딩**(+25.7%p, 논문에서 가장 큰 단일 개선 요인)을
  실제 라이브 타이핑 경로에 연결하면, `KNOWN_WORDS`/`KNOWN_SUFFIXES`
  같은 고정 규칙 없이도 조사·문맥 오류 상당수가 통계적으로 해소될 가능성.
  다만 `SentenceBuffer`가 실제 IME 입력 흐름(글자 단위로 즉시 커밋되는
  구조)과 맞물리게 재설계가 필요함.
- **N-gram 모델**(+6.7%p) — 짧은 문구 위주인 지금 예외 규칙 방식보다
  일반화가 잘 될 가능성.
- 논문 7.3절이 제안한 "실제 일본어 텍스트 코퍼스(위키피디아 등) 통합"은
  1.4절에서 확인했듯 **한 번도 실제로 완료된 적이 없습니다** —
  `data/hj-ime-large.db`는 시도의 흔적만 남은 빈 파일이었습니다. 이번에
  다시 시도한다면, 크롤링한 원문을 저장소에 커밋하지 말고(용량 문제)
  빌드 시점에 외부에서 내려받는 파이프라인으로 설계하는 게 안전합니다.
